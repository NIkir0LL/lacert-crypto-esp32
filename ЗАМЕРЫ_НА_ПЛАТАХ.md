# Замеры производительности прошивки на платах

English version: [`BOARD_MEASUREMENTS.md`](BOARD_MEASUREMENTS.md).

Методика измерения криптографических операций LACERT на реальных устройствах
ESP32 и проверки эффекта оптимизации ECDSA-контекста. Сборка под esp32c6
проверена из чистого клона репозитория; цифры на плате снимаются по шагам ниже.

## Проверено до замеров на плате

Независимо от целевой платы подтверждено:

- **Криптомодуль компилируется против mbedTLS 3.6.3** — той же версии, что
  входит в ESP-IDF v5.4. Библиотека собрана из исходников, `main/lacert_crypto.c`
  скомпилирован вместе с ML-KEM (PQClean) и BLAKE3, без ошибок и предупреждений.
- **Обе ветки переключателя дают верные подписи** — по 50 подписей двумя
  разными ключами, каждая проверена независимым контекстом (как это делает
  шлюз). Расхождений нет.
- **Полный протокол работает** — отладочный клиент из `linux-debug/` (тот же
  `lacert_wire.c` и `lacert_client.c`, что идут на плату) отработал против
  Go-шлюза: регистрация, рукопожатие, телеметрия, ротация ключа, ответы на
  проверку целостности прошивки — без ошибок.

Замеры на плате нужны, чтобы измерить поведение под Xtensa/RISC-V и аппаратным
ГСЧ ESP32, отличными от базовой платформы.

---

## Переключатель оптимизации ECDSA

В начале `main/lacert_crypto.c`:

```c
#define LACERT_SHARED_CRYPTO_CTX 1
```

- **1** — генератор случайных чисел и параметры кривой P-256 создаются один
  раз и переиспользуются (оптимизация);
- **0** — исходное поведение: контекст создаётся заново на каждую подпись.

Обе ветки в одной прошивке, переключаются одной цифрой — так их можно измерить
на одной плате в одинаковых условиях, иначе разница может оказаться следствием
разных сборок, а не самой оптимизации.

### Контекст

На mbedTLS 2.28 оптимизация давала выигрыш в 2,6 раза. На mbedTLS 3.6 (в составе
ESP-IDF v5.4) разницы нет: 348,2 мкс против 347,4 мкс по медиане пятнадцати
прогонов на x86-64. На ESP32 источник энтропии другой — аппаратный генератор
вместо подсистемы ОС, поэтому результат может отличаться, и это проверяется
только на плате.

## Порядок замеров

Для каждой платы (**XIAO ESP32-C6** и **XIAO ESP32-S3** — различаются наличием
аппаратного ускорителя ECC).

### 1. Подготовка

```bash
cd firmware
# заполнить в main/main.c: Wi-Fi, IP шлюза, LACERT_DEVICE_ID (уникальный!)
rm -rf build sdkconfig
idf.py set-target esp32c6        # или esp32s3
idf.py build
```

Проверить, что применилась оптимизированная конфигурация:

```bash
grep -E 'COMPILER_OPTIMIZATION|MBEDTLS_HARDWARE' sdkconfig
```

Если стоит `-Og` вместо `-Os` — замеры будут бессмысленны: отладочная сборка
на порядок медленнее.

### 2. Замер с оптимизацией (значение 1)

```bash
idf.py -p /dev/ttyACM0 flash monitor
```

Из вывода блока «МИКРОБЕНЧМАРК КРИПТОГРАФИИ» снять всю таблицу. Достаточно
одного прогона: операции миллисекундные, шум влияет слабо.

### 3. Замер без оптимизации

Поменять значение на `0` в `main/lacert_crypto.c`, затем пересобрать и залить
(пересборка с нуля не нужна — меняется один файл):

```bash
idf.py build
idf.py -p /dev/ttyACM0 flash monitor
```

### 4. Повторить для второй платы

## Формат результатов

Снимаются четыре таблицы микробенчмарка: C6 со значением 1, C6 со значением 0,
S3 со значением 1, S3 со значением 0. Блок имеет вид:

```
==================================================
МИКРОБЕНЧМАРК КРИПТОГРАФИИ (без сети)
плата: ESP32-C6 (RISC-V, 160 МГц)
--------------------------------------------------
  ECDSA P-256 keypair             12.25 мс   (5 прогонов)
  ECDSA P-256 sign                25.76 мс   (20 прогонов)
  ...
==================================================
```

По ним оценивается эффект оптимизации на каждой плате.

## Важные напоминания

**Уникальный `LACERT_DEVICE_ID` каждой плате.** Шлюз хранит один набор ключей
на идентификатор; вторая плата с тем же именем не пройдёт рукопожатие.

**Перепрошивка меняет SHA-256 образа.** Шлюз держит эталонный хеш с момента
регистрации, поэтому после перезаливки проверка целостности провалится и
устройство будет отозвано — это правильное поведение, но при частых
перепрошивках мешает. Перед перезаливкой убрать запись устройства:

```bash
psql "$LACERT_PG_DSN" -c "DELETE FROM devices WHERE device_id='xiao-c6';"
```

Либо, если шлюз работает без базы, перезапустить его.

**Порты:** XIAO — `/dev/ttyACM0`, ESP32-S3-DevKitC-1 — `/dev/ttyUSB0`.

**При смене цели** (`esp32c6` ↔ `esp32s3`) обязательно `rm -rf build sdkconfig`,
иначе сборка подхватит настройки предыдущей платы.

## Примечание о `bench_host`

Для сборки хостового бенчмарка (`MEASUREMENTS.md`, раздел 6.2) нужен флаг `-I.`,
иначе сборка падает на отсутствующем `esp_random.h`. В `linux-debug/` есть
заглушка `esp_random.h` (случайность из `getrandom(2)`, при недоступности — из
`/dev/urandom`), позволяющая воспроизвести замеры на базовой платформе.
