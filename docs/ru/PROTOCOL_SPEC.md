# LACERT — спецификация протокола (для реализации клиента на ESP32)

Этот документ описывает **байт-в-байт** формат протокола LACERT, чтобы клиент
на C (прошивка ESP32) точно совпал с реализацией шлюза на Go. Любое
расхождение хотя бы в одном байте (порядок полей, endianness, строка-сепаратор)
приведёт к тому, что рукопожатие или проверка подписи не пройдут.

Источник истины — код шлюза в `internal/wire`, `internal/crypto`. Здесь всё
извлечено из него.

---

## 0. Общие соглашения

- **Endianness:** все целочисленные длины и счётчики — **big-endian (BE)**.
- **Криптопримитивы:**
  - Обмен ключами: **ML-KEM-1024** (Kyber, финал NIST).
  - Подпись устройства: **ECDSA P-256** (рекомендуется; см. про SLH-DSA ниже).
  - Хеш/вывод ключей: **BLAKE3** (выход 32 байта).
  - Шифрование данных: **ChaCha20-Poly1305**.
  - Хеш прошивки: **SHA-256**.
- **Размеры (байты):**
  - ML-KEM-1024 public key: **1568**
  - ML-KEM-1024 ciphertext: **1568**
  - ML-KEM-1024 shared secret: **32**
  - Сессионный ключ (BLAKE3 выход): **32**
  - ECDSA P-256 public key (несжатый): **65** (`0x04 || X(32) || Y(32)`)
  - ECDSA P-256 подпись: **ASN.1 DER** (переменная длина, ~70–72 байта) —
    **НЕ** сырой формат r‖s!
  - ChaCha20-Poly1305 nonce: **12**, тег аутентификации: **16**
  - SHA-256 хеш: **32**
  - Handshake nonce (Msg1/Msg2): **32**
  - Firmware challenge: **64**

> **Важно про ECDSA-подпись.** Go подписывает через `ecdsa.SignASN1`, то есть
> результат — это структура ASN.1 DER (`SEQUENCE { INTEGER r, INTEGER s }`), а
> НЕ конкатенация r‖s. На ESP32 mbedTLS по умолчанию тоже выдаёт ASN.1 DER
> (`mbedtls_ecdsa_write_signature`), так что форматы совпадут. Не используйте
> «raw»-режим.

## 1. Формат кадра (транспорт поверх TCP)

Каждое сообщение оборачивается в кадр:

```
+-----------------+--------+------------------+
| длина payload   | тип    | payload          |
| 4 байта BE      | 1 байт | N байт           |
+-----------------+--------+------------------+
```

- Длина payload — `uint32` BE (не включает 5 байт заголовка).
- Максимальный размер кадра: **65536** байт (защита от мусора на входе).

**Типы сообщений (1 байт):**

| Тип | Значение | Направление | Назначение |
|-----|----------|-------------|------------|
| `TypeHandshakeMsg1` | 1 | устройство → шлюз | начало рукопожатия |
| `TypeHandshakeMsg2` | 2 | шлюз → устройство | ответ рукопожатия |
| `TypeHandshakeMsg3` | 3 | устройство → шлюз | подтверждение (подпись) |
| `TypeRotation` | 4 | — | (устаревшая неатомарная ротация, не используется) |
| `TypeData` | 5 | устройство → шлюз | зашифрованная телеметрия |
| `TypeFirmwareChallenge` | 6 | шлюз → устройство | запрос проверки прошивки |
| `TypeFirmwareResponse` | 7 | устройство → шлюз | ответ проверки прошивки |
| `TypeError` | 8 | шлюз → устройство | текстовая ошибка перед разрывом |
| `TypeRotationV2` | 9 | шлюз → устройство | атомарная ротация ключа |
| `TypeRotationAck` | 10 | устройство → шлюз | подтверждение ротации |

### Кадрирование полей внутри payload

Поля переменной длины внутри payload предваряются 2-байтовой BE-длиной:

```
putFramed(data): [uint16 BE: len(data)][data...]
```

Поля фиксированной длины (nonce 32 байта и т.п.) пишутся как есть, без префикса.

## 2. Офлайн-регистрация (до подключения)

Перед первым подключением устройство должно быть зарегистрировано на шлюзе
(«зашивка efuse» в реальной системе). Регистрация происходит через REST, а не
по TCP-протоколу. Тело запроса `POST /api/v1/devices`:

```json
{
  "device_id":         "<строка>",
  "identity_pub_hex":  "<hex ECDSA P-256 pub, 65 байт → 130 hex>",
  "kem_pub_hex":       "<hex ML-KEM-1024 pub, 1568 байт>",
  "firmware_hash_hex": "<hex SHA-256 прошивки, 32 байта>",
  "checksum":          "<контрольная сумма серийника>",
  "sig_algorithm":     "ecdsa-p256"
}
```

Заголовок: `Authorization: Bearer <admin-token>` (если шлюз защищён токеном).

Устройство хранит: свой приватный ключ ML-KEM-1024, приватный ключ ECDSA
(«efuse»), публичный ключ ML-KEM-1024 шлюза (получается через `GET
/api/v1/gateway`, поле `kem_pub_hex`).

## 3. Рукопожатие (Noise_XX-подобное, 3 сообщения)

### 3.1 Msg1 (устройство → шлюз), тип 1

Payload:

```
putFramed(device_id)          // [uint16 len][строка device_id]
nonce                         // 32 байта (случайные)
putFramed(identity_pub)       // [uint16 len][ECDSA pub, 65 байт]
```

Устройство генерирует случайный `nonce` (32 байта) и запоминает Msg1 целиком —
он понадобится для транскрипта.

### 3.2 Msg2 (шлюз → устройство), тип 2

Payload:

```
putFramed(kem_ciphertext)     // [uint16 len][ML-KEM ciphertext, 1568 байт]
gateway_nonce                 // 32 байта
```

Шлюз инкапсулировал общий секрет под публичным ML-KEM-ключом **устройства**.
Устройство **декапсулирует** секрет своим приватным ML-KEM-ключом:

```
shared_secret = ML-KEM-1024.Decapsulate(dev_kem_priv, kem_ciphertext)  // 32 байта
```

### 3.3 Вывод сессионного ключа K0

Оба вычисляют одинаково. Сначала **транскрипт** (BLAKE3 от канонических байт
Msg1 и Msg2):

```
msg1_bytes = putFramed(device_id) || nonce || putFramed(identity_pub)
msg2_bytes = putFramed(kem_ciphertext) || gateway_nonce

transcript = BLAKE3( msg1_bytes || msg2_bytes )        // 32 байта
```

> Внимание: `msg1_bytes`/`msg2_bytes` для транскрипта используют то же
> 2-байтовое кадрирование `putFramed`, что и на проводе. Порядок полей строго
> как выше.

Затем K0:

```
K0 = BLAKE3( shared_secret || transcript || "lacert_handshake_v1" )   // 32 байта
```

Строка-сепаратор — ASCII `lacert_handshake_v1` (без завершающего нуля).

### 3.4 Msg3 (устройство → шлюз), тип 3

Устройство подтверждает владение приватным ключом подписи и совпадение K0.
Сначала «значение подтверждения»:

```
confirm = BLAKE3( transcript || "confirm" || K0 )      // 32 байта
```

Строка-сепаратор — ASCII `confirm`. Затем устройство подписывает `confirm`:

```
signature = ECDSA_P256_sign( dev_identity_priv, confirm )
```

где `ECDSA_P256_sign(m)` = `ASN1( SignASN1( SHA-256(m) ) )`. То есть:
1. `digest = SHA-256(confirm)` (32 байта),
2. подпись `digest` ключом ECDSA, результат в формате **ASN.1 DER**.

Payload Msg3:

```
putFramed(signature)          // [uint16 len][ECDSA DER signature]
```

Шлюз проверяет подпись зарегистрированным публичным ключом устройства. Если
верна — сессия установлена, текущий ключ = K0.

---

> **Тип 4 (устаревший, не используется).** В ранней схеме тип 4 обозначал
> неатомарную ротацию ключа. Она признана небезопасной (кадр ничем не
> аутентифицирован и не сдвигает счётчик итераций) и из протокола убрана. Шлюз
> по-прежнему распознаёт тип 4 на входе, но **отвергает** его; штатные
> устройства и эмулятор такой кадр не отправляют. Актуальная ротация — атомарная,
> типы 9 и 10 (см. раздел 5). Значение 4 зарезервировано и не переиспользуется.

## 4. Передача данных (телеметрия)

### 4.1 Data (устройство → шлюз), тип 5

Полезные данные шифруются текущим сессионным ключом Ki:

```
nonce (12 байт) = random(8) || uint32_BE(seq_num)
ciphertext = ChaCha20Poly1305_seal(Ki, nonce, plaintext)   // включает 16-байтовый тег
```

- Старшие 8 байт nonce — случайные, младшие 4 байта — счётчик пакетов `seq_num`
  (BE). Это гарантирует уникальность nonce в рамках одного ключа.
- `plaintext` — строка телеметрии (например `temperature=25.3; seq=42`).

Payload:

```
putFramed(nonce)              // [uint16=12][nonce 12 байт]
putFramed(ciphertext)         // [uint16 len][ciphertext+тег]
```

Максимальный размер plaintext ограничен (см. `MaxPayloadSize` в коде).

## 5. Ротация ключей (атомарная, инициирует ШЛЮЗ)

Ротацию инициирует **только шлюз** (у устройства нет своего таймера ротации —
оно только отвечает). Устройство получает `TypeRotationV2`, применяет новый
ключ и отвечает `TypeRotationAck`.

### 5.1 RotationV2 (шлюз → устройство), тип 9

Payload:

```
iteration                     // 8 байт uint64 BE (номер новой итерации)
putFramed(kem_ciphertext)     // [uint16 len][ML-KEM ciphertext, 1568 байт]
```

Устройство декапсулирует свежий секрет `mi` своим приватным ML-KEM-ключом:

```
mi = ML-KEM-1024.Decapsulate(dev_kem_priv, kem_ciphertext)   // 32 байта
```

### 5.2 Вывод следующего ключа

```
next_key = BLAKE3( current_key(32) || mi(32) || uint64_BE(iteration) || "rotate_v1" )
```

- `current_key` — действующий сессионный ключ (32 байта).
- Строка-сепаратор — ASCII `rotate_v1`.
- Итерация — тот же `iteration` из сообщения (8 байт BE).

Устройство применяет `next_key` как новый текущий ключ.

### 5.3 RotationAck (устройство → шлюз), тип 10

Payload:

```
iteration                     // 8 байт uint64 BE (та же итерация)
```

Шлюз, получив ACK с правильным номером итерации, коммитит новый ключ у себя.
Если ACK не пришёл за `RotationAckTimeout` (по умолчанию 5с) — шлюз откатывает
ротацию; после нескольких неудач подряд устройство отзывается.

> **Проверка номера итерации.** Устройство должно принимать только
> `iteration == current_iteration + 1`. Меньший или равный — это повтор/replay
> (отклонить). Больший с пропуском — рассинхронизация (отклонить).

---

## 6. Проверка целостности прошивки

Шлюз периодически проверяет, что прошивка устройства не подменена.

### 6.1 FirmwareChallenge (шлюз → устройство), тип 6

Payload:

```
putFramed(challenge)          // [uint16=64][случайные 64 байта]
```

### 6.2 FirmwareResponse (устройство → шлюз), тип 7

Устройство вычисляет SHA-256 своего образа прошивки и подписывает
`challenge || firmware_hash`:

```
firmware_hash = SHA-256(firmware_image)                      // 32 байта
to_sign = challenge(64) || firmware_hash(32)                 // 96 байт
signature = ECDSA_P256_sign( dev_identity_priv, to_sign )
          = ASN1( SignASN1( SHA-256(to_sign) ) )
```

Payload:

```
firmware_hash                 // 32 байта (без putFramed, фиксированный размер!)
putFramed(signature)          // [uint16 len][ECDSA DER signature]
```

> Внимание: `firmware_hash` пишется как фиксированные 32 байта БЕЗ 2-байтового
> префикса длины, сразу в начало payload. Подпись — с префиксом.

Шлюз сверяет `firmware_hash` с эталонным (из регистрации) и проверяет подпись.
Ответ должен прийти в окне `firmwareResponseValidity` (по умолчанию 15с), иначе
challenge считается устаревшим и отклоняется.

---

## 7. Ошибки (шлюз → устройство), тип 8

Payload — просто UTF-8 строка с текстом ошибки (например `device revoked`).
Отправляется перед закрытием соединения. Кадрирования полей нет — весь payload
это строка.

---

## 8. Последовательность работы устройства (сводка)

```
1. (однократно) Регистрация через REST + получение ML-KEM pub шлюза.
2. Подключиться по TCP к шлюзу (порт 7700 по умолчанию).
3. Отправить Msg1 → получить Msg2 → decapsulate → вычислить K0 →
   отправить Msg3. Сессия установлена.
4. В цикле:
   - отправлять Data (телеметрию), шифруя текущим ключом;
   - слушать входящие кадры:
       * RotationV2  → применить новый ключ, ответить RotationAck;
       * FirmwareChallenge → ответить FirmwareResponse;
       * Error → залогировать и переподключиться/остановиться.
```

---

## 9. Про выбор схемы подписи

Система поддерживает обе схемы (`sig_algorithm`: `ecdsa-p256` или `slh-dsa`),
но для ESP32 **настоятельно рекомендуется ECDSA P-256**:

- **ECDSA P-256**: подпись 22,2 мс на ESP32-C6 и 170,2 мс на ESP32-S3, размер
  ~70 байт. Разница между платами объясняется аппаратным ускорителем
  эллиптических кривых: он есть у ESP32-C6 и отсутствует у ESP32-S3, поэтому на
  S3 операция выполняется полностью программно (mbedTLS).
- **SLH-DSA**: подпись — сотни миллисекунд даже на сервере (примерно в 11 000
  раз медленнее ECDSA), размер 7856 байт. На ESP32 неприемлемо ни по времени,
  ни по памяти.

Это подтверждено измерениями (см. `MEASUREMENTS.md`). ML-KEM обеспечивает
постквантовую стойкость на уровне обмена ключами; подпись остаётся классической.

Стоит учитывать, что на ESP32-S3 подпись оказывается самой дорогой операцией
протокола — дороже постквантовой декапсуляции ML-KEM (170,2 мс против 21,1 мс).

### Почему подпись осталась классической

Из двух рассмотренных схем постквантовую стойкость даёт только SLH-DSA: её
безопасность опирается на свойства хеш-функций, которые алгоритм Шора не
затрагивает. ECDSA стоит на задаче дискретного логарифма и квантовым
противником взламывается. Замеры показали, что цена постквантовой подписи для
ESP32 неприемлема, поэтому принято осознанное разделение:

- **обмен ключами — постквантовый** (ML-KEM-1024);
- **подпись — классическая** (ECDSA P-256).

Практическое следствие: противник, располагающий квантовым компьютером, не
сможет восстановить сессионные ключи из перехваченного трафика, но сможет
подделать подпись устройства. Для сценария, в котором защищается прежде всего
конфиденциальность длительно хранимых данных («перехватить сейчас, расшифровать
потом»), такое разделение оправдано; для сценария, где критична именно
неподделываемость подписи, потребуется схема уровня SLH-DSA и, соответственно,
более производительное устройство.

### Рассмотренная альтернатива: Ed25519

В ходе работы как замена ECDSA рассматривался Ed25519. Схема не реализовывалась
самостоятельно — использовалась стандартная библиотека Go (`crypto/ed25519`),
поэтому корректность самого алгоритма обеспечивается её авторами. Со стороны
проекта проверено встраивание: подпись и проверка на сгенерированной паре
ключей, отклонение подделанного сообщения и прохождение полного рукопожатия
(`TestEd25519RoundTrip` и `TestHandshakeEd25519` в `internal/crypto`). Замеры на
серверной платформе дали заметный выигрыш — подпись быстрее на 23 %, а объём
выделяемой памяти меньше в 99 раз (см. `MEASUREMENTS.md`, раздел 3.2).

Тем не менее от замены решено отказаться:

1. **Постквантовой стойкости она не добавляет.** Ed25519 опирается на ту же
   задачу дискретного логарифма, что и ECDSA, и уязвим к алгоритму Шора в
   равной мере. Основную задачу работы такая замена не решает.
2. **Требуется четвёртая внешняя зависимость.** ECDSA входит в mbedTLS, то есть
   уже присутствует в ESP-IDF (см. раздел 10), тогда как Ed25519 пришлось бы
   подключать отдельной библиотекой.
3. **На ESP32-C6 замена ухудшила бы результат.** Аппаратный ускоритель этой
   платы разгоняет операции ECDSA примерно в 7,7 раза (`MEASUREMENTS.md`,
   наблюдение 8); для Ed25519 аппаратной поддержки нет.

Реализация Ed25519 сохранена в ядре шлюза (`internal/crypto`) как инструмент
сравнительных измерений, но в прошивке и в протоколе не используется.

## 10. Криптобиблиотеки на C для ESP32 (ориентир)

| Примитив | Откуда брать |
|----------|--------------|
| ECDSA P-256 | mbedTLS (встроен в ESP-IDF; аппаратное ускорение только на ESP32-C6) |
| ChaCha20-Poly1305 | mbedTLS (встроен) |
| SHA-256 | mbedTLS / ROM (аппаратный) |
| ML-KEM-1024 | PQClean — подключён как компонент `firmware/components/ml_kem` |
| BLAKE3 | официальная C-реализация BLAKE3-team/BLAKE3 |

Дописывать с нуля надо только ML-KEM-1024 и BLAKE3 — остальное уже в ESP-IDF.
