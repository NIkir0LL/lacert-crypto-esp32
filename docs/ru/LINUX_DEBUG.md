# Linux-отладка C-клиента LACERT

Здесь собирается **Linux-версия** C-клиента прошивки — для отладки протокола
против работающего Go-шлюза **без железа ESP32**. Это ускоряет разработку:
быстрая пересборка (секунды), обычный отладчик gdb, санитайзеры памяти.

Клиент использует **те же** `../main/lacert_wire.c` и `../main/lacert_client.c`,
что и прошивка ESP32 — то есть проверяется ровно та логика протокола, что
поедет на плату. Отличается только крипто-слой: здесь OpenSSL (файл
`lacert_crypto_linux.c`) вместо mbedTLS, но криптопримитивы те же (ECDSA P-256,
ChaCha20-Poly1305, SHA-256, ML-KEM-1024, BLAKE3).

## Проверенная совместимость

Ключевые крипто-библиотеки проверены на **точное совпадение** с реализацией
Go-шлюза (иначе рукопожатие не прошло бы):

- **ML-KEM-1024**: PQClean (C) даёт тот же shared secret, что circl (Go).
- **BLAKE3**: официальная C-реализация даёт тот же хеш, что zeebo/blake3 (Go).
- **ChaCha20-Poly1305 / ECDSA**: совместимы (телеметрия, зашифрованная на C,
  расшифровывается шлюзом; подпись рукопожатия принимается).

End-to-end тест пройден: C-клиент регистрируется, проводит рукопожатие и шлёт
телеметрию — шлюз всё принимает и расшифровывает корректно.

## Установка зависимостей (Ubuntu/Debian)

```bash
sudo apt-get update
sudo apt-get install -y build-essential libssl-dev
git clone https://github.com/BLAKE3-team/BLAKE3 ~/BLAKE3
git clone https://github.com/PQClean/PQClean ~/PQClean
```

## Сборка

Путь ниже указан для монорепозитория `lacert`. В репозитории
`lacert-crypto-esp32` отладочный клиент лежит в корне — там достаточно
`cd linux-debug`.

```bash
cd firmware/linux-debug
bash build_linux.sh
```

(при нестандартном расположении библиотек: `BLAKE3_DIR=... PQCLEAN_DIR=... bash build_linux.sh`)

## Запуск против шлюза

Шлюз должен быть запущен. Затем:

```bash
LACERT_ADMIN_TOKEN=<ваш-токен> LACERT_DEVICE_ID=linux-test-1 \
  ./lacert-client <host> <http_port> <tcp_port>
```

Например, против локального шлюза:

```bash
LACERT_ADMIN_TOKEN=41b1751e... ./lacert-client 127.0.0.1 8080 7700
```

Клиент:
1. сгенерирует ключи ECDSA + ML-KEM;
2. зарегистрируется через REST (POST /api/v1/devices);
3. получит публичный ML-KEM шлюза;
4. проведёт рукопожатие;
5. отправит 5 пакетов телеметрии.

Успешный вывод заканчивается строкой «протокол работает». Расшифрованную
телеметрию можно увидеть на дашборде или через
`GET /api/v1/telemetry?device_id=<id>`.

## Файлы

- `lacert_crypto_linux.c` — крипто на OpenSSL + PQClean + BLAKE3 (только Linux).
- `main_linux.c` — точка входа: генерация ключей, регистрация, рукопожатие,
  отправка данных.
- `build_linux.sh` — скрипт сборки.
- `esp_random.h` — заглушка ESP32-API генератора случайных чисел для сборки на
  хосте. Нужна только для `bench_host`, который компилирует `../main/lacert_crypto.c`
  напрямую; случайность берётся из `getrandom(2)` либо `/dev/urandom`. В прошивке
  используется настоящий заголовок из состава ESP-IDF.

Для ESP32 эти файлы НЕ используются — там `../main/lacert_crypto.c` (mbedTLS) и
`../main/main.c`.

## Проверка ротации и прошивки (полный протокол)

Клиент живёт `LACERT_RUN_SECONDS` секунд (по умолчанию 60), слушает шлюз и
отвечает на инициированные им ротации ключа и проверки прошивки, параллельно
отправляя телеметрию. Чтобы увидеть ротацию/прошивку быстро, запустите шлюз с
ускоренными интервалами:

```bash
# на стороне шлюза:
LACERT_ROTATION_INTERVAL=6s LACERT_FIRMWARE_INTERVAL=5s LACERT_ROTATION_CHECK_PERIOD=1s ./gatewayd
# на стороне клиента:
LACERT_RUN_SECONDS=25 ./lacert-client 127.0.0.1 8080 7700
```

В выводе клиента появятся строки `[ротация] применён новый ключ` и
`[прошивка] ответил на challenge`, а в конце — сводка. Это подтверждает, что
C-реализация ротации (вывод следующего ключа через BLAKE3) и проверки прошивки
(ECDSA-подпись challenge||hash) совместима со шлюзом.

## Приближение к реальной прошивке

main_linux.c доработан так, чтобы моделировать поведение реального устройства:

1. **Постоянные ключи.** При первом запуске ключи (ECDSA + ML-KEM)
   генерируются и сохраняются в файл (по умолчанию `lacert_device.keys`,
   переопределяется `LACERT_KEYFILE`). При последующих запусках — загружаются,
   поэтому шлюз узнаёт «то же устройство» после перезапуска (как NVS/efuse на
   ESP32). Повторная регистрация с теми же ключами шлюзом игнорируется — это
   штатно.
2. **Реальный хеш прошивки.** firmware_hash = SHA-256 собственного бинарника
   (на ESP32 — SHA-256 образа прошивки). Один и тот же между запусками.
3. **Переподключение.** При разрыве связи клиент ждёт 2с и переподключается,
   заново запрашивая публичный ключ шлюза и повторяя рукопожатие — как реальное
   устройство после потери сети.

Что осталось «линуксового» (заменяется на ESP32): крипто на OpenSSL →
mbedTLS + компоненты; файловое хранилище ключей → NVS/efuse; POSIX-сокеты →
WiFi + lwIP (API сокетов тот же).

### Пример: повторный запуск = то же устройство

```bash
# первый раз — регистрируется:
LACERT_KEYFILE=dev1.keys ./lacert-client 127.0.0.1 8080 7700
# второй раз — узнаётся по тем же ключам:
LACERT_KEYFILE=dev1.keys ./lacert-client 127.0.0.1 8080 7700
```
