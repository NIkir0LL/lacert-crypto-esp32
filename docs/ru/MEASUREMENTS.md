# Методика измерений и результаты

Раздел описывает, как получены все числовые данные, приведённые в работе.
Измерения выполнены самостоятельно на собственном стенде; ни одно значение не
заимствовано из литературы или документации производителей. Приведённого здесь
описания достаточно, чтобы воспроизвести любой замер.

### 1.3. Конфигурация сборки прошивки

Параметры сборки существенно влияют на результат, поэтому фиксируются явно
(файл `firmware/sdkconfig.defaults`):

```
CONFIG_COMPILER_OPTIMIZATION_SIZE=y     # -Os (а не отладочный -Og по умолчанию)
CONFIG_MBEDTLS_HARDWARE_MPI=y           # аппаратный блок больших чисел
CONFIG_MBEDTLS_HARDWARE_SHA=y           # аппаратный SHA-256
CONFIG_MBEDTLS_ECP_NIST_OPTIM=y         # оптимизированная арифметика для P-256
CONFIG_MBEDTLS_HARDWARE_ECC=y           # только ESP32-C6: ускоритель ECC
```

**Обоснование.** ESP-IDF по умолчанию собирает проект с `-Og` (отладочная
сборка). На ней криптографические операции выполняются в разы медленнее, чем в
реальной прошивке, и приведённые цифры характеризовали бы не алгоритм, а режим
отладки. Все измерения в работе получены на оптимизированной сборке; обе платы
собраны с одинаковыми параметрами, поэтому сравнение архитектур корректно.

## 2. Методика измерения криптографических операций

### 2.1. Что измеряется

Измеряется время выполнения отдельных криптографических примитивов **без сетевого
обмена**. Это принципиально: полное время рукопожатия включает три обмена
пакетами по Wi-Fi, задержка которых составляет десятки миллисекунд, нестабильна
и от кристалла не зависит. Сравнивать по ней архитектуры некорректно.

Инструмент: режим `LACERT_BENCH` в прошивке (`firmware/main/main.c`). Запускается
автоматически при включении платы, до подключения к сети, и выводит результаты в
последовательный порт.

### 2.2. Порядок замера

Операции разделены на две группы, поскольку они различаются по стоимости на
четыре порядка:

**Тяжёлые операции** (ECDSA, ML-KEM — единицы и сотни миллисекунд).
Каждая итерация замеряется отдельно, результаты суммируются и делятся на число
итераций (20; для генерации ключевых пар — 5). Между итерациями выполняется
`vTaskDelay(1)`: без этого задача удерживает процессорное ядро дольше тайм-аута
сторожевого таймера (одна подпись ECDSA на ESP32-S3 занимает около 170 мс,
двадцать подряд — 3,5 с), и происходит аварийный сброс. Пауза находится **между**
замерами и в измеряемый интервал не входит.

**Лёгкие операции** (BLAKE3, SHA-256, ChaCha20-Poly1305 — десятки микросекунд).
Замеряется цикл целиком (200 итераций), одним обращением к таймеру на весь цикл.
Поитерационный замер здесь недопустим: два вызова функции времени стоят несколько
микросекунд, что при длительности операции в десятки микросекунд завышает
результат на десятки процентов. Экспериментальная проверка: BLAKE3 при
поитерационном замере показывал 42,9 мкс, при замере цикла — 17,9 мкс.

Источник времени на плате — `esp_timer_get_time()` (аппаратный таймер,
разрешение 1 мкс).

### 2.3. Сопоставимость с базой сравнения

Значения для x86-64 получены **той же реализацией и той же методикой**:
программа `firmware/linux-debug/bench_host.c` собирается из тех же исходных
файлов криптографического ядра (`lacert_crypto.c`, PQClean, BLAKE3), что и
прошивка. Благодаря этому расхождения между платформами нельзя объяснить
различием библиотек — сравнивается один и тот же код.

## 3. Результаты: криптографические операции

Среднее время одной операции. Каждое значение — среднее по 20 запускам
(генерация ключевых пар — 5, лёгкие операции — 200).

| Операция | x86-64 | ESP32-S3 (Xtensa, 240 МГц) | ESP32-C6 (RISC-V, 160 МГц) |
|----------|--------|---------------------------|---------------------------|
| ECDSA P-256, подпись | 0,35 мс | 170,2 мс | **22,2 мс** |
| ECDSA P-256, генерация пары | 0,30 мс | 156,7 мс | **9,6 мс** |
| ML-KEM-1024, генерация пары | 0,11 мс | 16,7 мс | 15,1 мс |
| ML-KEM-1024, инкапсуляция | 0,11 мс | 18,4 мс | 16,0 мс |
| ML-KEM-1024, декапсуляция | 0,12 мс | 21,1 мс | 17,8 мс |
| BLAKE3, вывод ключа | 2,0 мкс | 18,0 мкс | 19,8 мкс |
| SHA-256 (96 байт) | 0,7 мкс | 60,1 мкс | 41,7 мкс |
| ChaCha20-Poly1305, шифрование | 12,4 мкс | 150,5 мкс | 210,6 мкс |
| Свободная память после инициализации | — | 252 КБ | 295 КБ |

Значения для плат сняты в конфигурации `LACERT_SHARED_CRYPTO_CTX = 1` (общий
криптографический контекст, см. раздел 3.4).

Размер подписи ECDSA P-256 в кодировании DER — переменный, 69–72 байта
(зависит от старших нулей в компонентах подписи).

### 3.1. Основные наблюдения

**1. Подпись ECDSA на ESP32-C6 выполняется в 7,7 раза быстрее, чем на
ESP32-S3**, несмотря на то, что C6 уступает по вычислительным характеристикам
(одно ядро против двух, 160 МГц против 240).

Причина не в производительности ядра, что подтверждается остальными строками
таблицы: на операциях ML-KEM платы сопоставимы (17,8 против 21,1 мс), а на
BLAKE3 и ChaCha20 ESP32-C6 даже **медленнее** ESP32-S3 — как и следует ожидать
при меньшей тактовой частоте. Ускорение наблюдается исключительно на операциях с
эллиптическими кривыми, для которых у ESP32-C6 имеется аппаратный ускоритель,
отсутствующий у ESP32-S3.

**Вывод:** при выборе микроконтроллера для задач с интенсивной криптографией
наличие профильного аппаратного ускорителя оказывается важнее тактовой частоты и
числа ядер.

**2. Производительность ML-KEM-1024 слабо зависит от платформы** (16–21 мс на
обеих платах). Аппаратных ускорителей решёточной криптографии в исследуемых
кристаллах нет, операции выполняются полностью программно. Следовательно,
переход на постквантовый обмен ключами обходится примерно одинаково на любом
кристалле этого класса, тогда как стоимость подписи существенно зависит от
конкретной модели.

**3. Соотношение стоимости на микроконтроллере и на сервере различно для разных
примитивов.** Подпись ECDSA на ESP32-S3 медленнее серверной в ~490 раз, а
операции ML-KEM — в ~170 раз. То есть на микроконтроллере постквантовый обмен
ключами «дешевеет» относительно классической подписи, что дополнительно
поддерживает выбранную архитектуру (постквантовый KEM + ECDSA).

### 3.2. Сравнение алгоритмов подписи на серверной платформе

Отдельная серия замеров — сравнение трёх схем подписи в реализации шлюза (Go).
В отличие от раздела 3, где сравнивались платформы, здесь сравниваются
**алгоритмы между собой** на одной машине.

**Стенд и методика.** AMD Ryzen 7 5700X, Windows, Go 1.22, штатный механизм
бенчмарков (`go test -bench`). Быстрые операции измерены по времени
(`-benchtime=2s`, порядка 10⁵ итераций) с тройным повтором (`-count=3`);
разброс между повторами не превысил 1,5 %. SLH-DSA измерена по числу итераций
(`-benchtime=10x`), поскольку одна подпись занимает около четверти секунды.

| Алгоритм | Стойкость к квантовому противнику | Подпись | Проверка | Генерация пары | Память на подпись |
|----------|-----------------------------------|---------|----------|----------------|-------------------|
| ECDSA P-256 | нет (дискретный логарифм) | 25,2 мкс | 56,7 мкс | 11,2 мкс | 6336 Б / 67 выделений |
| Ed25519 | нет (дискретный логарифм) | **19,5 мкс** | **45,1 мкс** | 16,0 мкс | **64 Б / 1 выделение** |
| SLH-DSA-SHA2-128s | **да** (стойкость хеш-функций) | 234,3 мс | 232,5 мкс | — | 15 670 Б / 37 выделений |

Колонка стойкости существенна для интерпретации остальных: ECDSA и Ed25519
опираются на одну и ту же задачу — дискретный логарифм в группе точек
эллиптической кривой, — и алгоритм Шора решает её за полиномиальное время,
обесценивая обе схемы одинаково. Различие кривых (P-256 против edwards25519) на
это не влияет. Постквантовую стойкость среди рассмотренных схем подписи даёт
только SLH-DSA.

Вспомогательные операции протокола на той же машине: генерация пары
ML-KEM-1024 — 36,5 мкс, инкапсуляция — 12,9 мкс, декапсуляция — 19,4 мкс,
шаг ротации ключа — 34,7 мкс, шифрование пакета — 0,83 мкс.

**Наблюдение 1. У SLH-DSA дорога только подпись.** Проверка занимает 232,5 мкс
и уступает ECDSA всего вчетверо, тогда как подпись — 234,3 мс, то есть в 9300
раз дороже ECDSA и **в тысячу раз дороже собственной проверки**.

Для рассматриваемой архитектуры эта асимметрия неблагоприятна: подпись
формирует устройство (микроконтроллер), а проверяет шлюз (сервер). Таким
образом, вся стоимость алгоритма приходится на наименее производительный узел
системы.

**Наблюдение 2. Ed25519 выигрывает у ECDSA умеренно по времени и радикально по
памяти.** Подпись быстрее на 23 %, проверка — на 20 %, однако объём выделяемой
памяти отличается в 99 раз, а число обращений к распределителю памяти — в 67
раз (одно против шестидесяти семи). Генерация ключевой пары у Ed25519, наоборот,
на 44 % медленнее, что практического значения не имеет: она выполняется
однократно за время жизни устройства.

На серверной платформе выигрыш по памяти несущественен, однако для
микроконтроллера с ограниченной кучей он определяет предсказуемость поведения:
меньше фрагментации и меньше риск отказа в выделении памяти.

Важная оговорка: **Ed25519 не является постквантовой схемой**. Приведённое
сравнение носит инженерный характер — оно отвечает на вопрос о стоимости
классической подписи, но не о стойкости к квантовому противнику. Замена ECDSA
на Ed25519 не приблизила бы систему к постквантовой защите: обе схемы стоят на
одной задаче и уязвимы к алгоритму Шора в равной мере. Итоговый выбор схемы
подписи и причины отказа от Ed25519 изложены в `PROTOCOL_SPEC.md`, раздел 9.

### 3.3. Стоимость выбора подписи на уровне протокола

Приведённые выше значения характеризуют изолированные операции. Практический
интерес представляет вопрос, во что выбор алгоритма обходится **всему
протоколу**. Для этого измерено полное рукопожатие (три сообщения, включая
обмен ML-KEM, вывод ключа и подтверждение) в двух вариантах:

| Вариант рукопожатия | Время | Память | Выделений |
|---------------------|-------|--------|-----------|
| с ECDSA P-256 | 122,8 мкс | 13 704 Б | 108 |
| с SLH-DSA-SHA2-128s | 235,2 мс | 28 228 Б | 92 |

Переход на SLH-DSA удорожает протокол **в 1914 раз**. Это соотношение
существенно ниже, чем 9300 для изолированной подписи, и точнее отражает
практические последствия выбора.

Примечательно, что SLH-DSA требует вдвое больше памяти, но меньше выделений
(92 против 108): подпись передаётся единым блоком в 7856 байт, тогда как ECDSA
распределяет работу по множеству мелких выделений.

**Структура рукопожатия с ECDSA.** Разложение полного времени по операциям:

| Операция | Время | Доля |
|----------|-------|------|
| Проверка подписи (шлюз) | 56,7 мкс | 46 % |
| Формирование подписи (устройство) | 25,2 мкс | 20 % |
| ML-KEM декапсуляция | 19,4 мкс | 16 % |
| ML-KEM инкапсуляция | 12,9 мкс | 10 % |
| BLAKE3, кадрирование, прочее | 8,8 мкс | 7 % |

**Наблюдение 3.** Операции с подписью составляют 67 % времени рукопожатия,
постквантовый обмен ключами — 26 %. Следовательно, узким местом протокола
является классическая подпись, а не постквантовая часть, — причём это
справедливо уже на серверной платформе, а не только на микроконтроллере, где
тот же эффект выражен сильнее (см. наблюдение 1 раздела 3.1).

Полученный результат подтверждает выбранную архитектуру: перенос постквантовой
стойкости на уровень обмена ключами не создаёт вычислительного узкого места, а
дальнейшая оптимизация протокола должна быть направлена на операции подписи.

### 3.4. Стоимость подготовки криптографического контекста

В исходной реализации прошивки генератор случайных чисел (CTR-DRBG) и параметры
кривой P-256 создавались заново при каждом вызове подписи. Проверялась гипотеза,
что вынос их в общий контекст, создаваемый однократно, сократит время операции.

Переключатель `LACERT_SHARED_CRYPTO_CTX` в `firmware/main/lacert_crypto.c`
позволяет собрать обе версии из одного исходного текста, что даёт возможность
измерить их на одной плате в одинаковых условиях.

| Плата | Операция | Контекст на вызов | Общий контекст | Выигрыш |
|-------|----------|-------------------|----------------|---------|
| ESP32-C6 | подпись | 25,70 мс | 22,20 мс | 13,6 % |
| ESP32-C6 | генерация пары | 12,40 мс | 9,63 мс | 22,3 % |
| ESP32-S3 | подпись | 172,48 мс | 170,23 мс | 1,3 % |
| ESP32-S3 | генерация пары | 158,04 мс | 156,67 мс | 0,9 % |

**Наблюдение 4. Экономия имеет фиксированную величину, а её относительный вес
зависит от производительности криптографии на кристалле.** В абсолютном
выражении подготовка контекста обходится в 1,4–3,5 мс на обеих платах, что
естественно: эта работа не зависит от скорости последующих вычислений. Однако на
ESP32-C6, где подпись занимает 22 мс благодаря аппаратному ускорителю, эти
миллисекунды составляют заметную долю, тогда как на ESP32-S3 с его 170 мс они
теряются в общем времени.

Различие в 1,3 % для ESP32-S3 сопоставимо с погрешностью измерения: контрольные
операции, не затронутые изменением, расходились между прогонами до 6,5 %
(SHA-256), тогда как остальные — в пределах 0,6 %. Для ESP32-C6 выигрыш в
13,6 % и 22,3 % лежит существенно выше уровня шума.

**Плата за оптимизацию.** Общий контекст постоянно занимает место в куче: 608
байт на ESP32-C6 и 448 байт на ESP32-S3. При 250–300 КБ свободной памяти это
несущественно.

Оптимизация оставлена включённой по умолчанию. Переключатель сохранён в коде,
поэтому решение обратимо и поддаётся повторной проверке.

**Отдельное замечание о серверной платформе.** На x86-64 при сборке против
mbedTLS 3.6 разница между двумя версиями отсутствует: 348,2 мкс против 347,4 мкс
по медиане пятнадцати прогонов. Выигрыш проявляется только на микроконтроллере,
где источником энтропии служит аппаратный генератор, а не подсистема
операционной системы. Это иллюстрирует, что оптимизации, измеренные на серверной
платформе, не переносятся на встраиваемую автоматически, и наоборот.

### 3.5. Сравнение с DTLS 1.2

Приведённые выше значения характеризуют LACERT сам по себе. Практический вопрос
— во что обходится стандартное решение той же задачи. В качестве базы сравнения
взят DTLS 1.2 в реализации mbedTLS: он предназначен для тех же сценариев
(защита датаграмм от встраиваемых устройств) и входит в состав ESP-IDF, то есть
не требует дополнительных зависимостей.

**Условия сопоставимости.** Обе стороны каждого протокола выполняются в одном
процессе и обмениваются через буферы в памяти: сеть исключена, иначе измерялась
бы задержка канала. Обе программы собраны одним компилятором и слинкованы с
одной сборкой mbedTLS 3.6.3.

Последнее существенно. Рукопожатие LACERT измерено также в реализации шлюза на
Go (раздел 3.3, 122,8 мкс), однако сравнивать это значение с DTLS нельзя: в Go
подпись P-256 выполняется ассемблерной реализацией за 25 мкс, тогда как та же
операция в mbedTLS занимает 365 мкс. Разница в 14 раз отражала бы выбор
библиотеки, а не свойства протоколов. Поэтому для сравнения написан отдельный
стенд, воспроизводящий криптографический путь рукопожатия LACERT на C с тем же
mbedTLS.

**Выбор режимов DTLS.** Взяты три, поскольку они дают разные гарантии:

- **DTLS-PSK** — общий секрет прошит заранее. Нижняя граница стоимости, но
  аутентификации на основе ключа конкретного устройства нет, и отзыв одного
  устройства требует смены ключа у всех.
- **DTLS-ECDHE-PSK** — то же плюс эфемерный обмен ключами.
- **DTLS-ECDHE-ECDSA со взаимной проверкой сертификатов** — ближайший аналог
  LACERT по свойствам: обе стороны доказывают подлинность, сеансовый ключ
  свежий.

Наборы шифров заданы явно. Без этого mbedTLS для «PSK» самостоятельно выбирает
ECDHE-PSK, и замер перестаёт быть нижней границей.

#### Результаты (x86-64, mbedTLS 3.6.3, медиана пяти прогонов)

| Протокол | Установление сессии | Трафик рукопожатия |
|----------|---------------------|--------------------|
| DTLS-PSK | 89,9 мкс | 408 байт |
| **LACERT** | **1846 мкс** | **1800 байт** |
| DTLS-ECDHE-PSK | 2816 мкс | 538 байт |
| DTLS-ECDHE-ECDSA (взаимная) | 5964 мкс | 1649 байт |

Разброс между прогонами: 3–10 %.

**Наблюдение 5. При сопоставимых гарантиях LACERT устанавливает сессию в 3,2
раза быстрее DTLS.** Сравнение корректно именно с вариантом ECDHE-ECDSA со
взаимной проверкой: только он, как и LACERT, аутентифицирует устройство по его
собственному ключу и вырабатывает свежий сеансовый ключ.

Причина преимущества видна в разложении рукопожатия LACERT:

| Операция | Время | Доля |
|----------|-------|------|
| Проверка подписи ECDSA (шлюз) | 1220,4 мкс | 66 % |
| Формирование подписи ECDSA (устройство) | 366,7 мкс | 20 % |
| ML-KEM-1024 декапсуляция (устройство) | 131,3 мкс | 7 % |
| ML-KEM-1024 инкапсуляция (шлюз) | 112,9 мкс | 6 % |
| BLAKE3, вывод ключа | 60,8 мкс | 3 % |

**Наблюдение 6. Постквантовый обмен ключами дешевле классического.** На ML-KEM
приходится 13 % рукопожатия, тогда как ECDHE в DTLS обходится примерно в
2700 мкс — это следует из разницы между DTLS-PSK и DTLS-ECDHE-PSK. То есть
замена ECDHE на ML-KEM-1024 не только даёт постквантовую стойкость, но и
сокращает время.

Различие в общей стоимости объясняется тем, что DTLS-ECDHE-ECDSA выполняет и
эфемерный обмен на эллиптической кривой, и две подписи с разбором сертификатов,
тогда как LACERT ограничивается одной подписью и инкапсуляцией.

#### Объём трафика — не в пользу LACERT

Рукопожатие LACERT занимает 1800 байт против 538 у DTLS-ECDHE-PSK. Причина —
шифротекст ML-KEM-1024 размером 1568 байт; это неустранимое свойство схемы.

| Сообщение | Размер | Содержимое |
|-----------|--------|------------|
| Msg1 | 113 байт | идентификатор, nonce 32, публичный ключ ECDSA 65 |
| Msg2 | 1602 байта | шифротекст ML-KEM 1568, nonce 32 |
| Msg3 | 73 байта | подпись ECDSA |
| Заголовки кадров | 12 байт | тип и длина, три сообщения |

Для устройства с автономным питанием это может весить больше, чем выигрыш во
времени: передача по радио обычно дороже вычислений. Смягчающее обстоятельство
— рукопожатие выполняется один раз за сессию, а последующая ротация ключей идёт
без нового обмена ML-KEM и добавляет к трафику лишь 32 байта на шаг.

#### Проверка на платах: ESP32-S3 и ESP32-C6

Тот же стенд выполнен на обеих платах, где клиент и сервер работают на одном
кристалле. Платы различаются наличием аппаратного ускорителя эллиптических
кривых: он есть у ESP32-C6 и отсутствует у ESP32-S3. Это делает сравнение
особенно показательным — ускоритель играет на стороне DTLS, поскольку
операциям ML-KEM аппаратной поддержки не предоставляет ни одна из плат.

| Протокол | ESP32-S3 | ESP32-C6 |
|----------|----------|----------|
| DTLS-ECDHE-PSK | 530,8 мс (407 Б) | 505,9 мс (407 Б) |
| DTLS-ECDHE-ECDSA (взаимная) | 1555,0 мс (1455 Б) | 649,1 мс (1452 Б) |

Режим DTLS-PSK в сборке mbedTLS из ESP-IDF недоступен, поэтому нижняя граница
не измерялась. Объём трафика отличается от значений на x86 (538 и 1649 байт):
сборка ESP-IDF включает иной набор расширений TLS.

**Наблюдение 7. Постквантовый обмен ключами дешевле классического на обеих
платах, причём на плате с аппаратным ускорителем — в большей степени.**
Стоимость ECDHE оценивается величиной DTLS-ECDHE-PSK, поскольку симметричная
часть рукопожатия пренебрежимо мала на её фоне.

| | ESP32-S3 | ESP32-C6 | x86-64 |
|---|---|---|---|
| ECDHE (классический обмен) | 530,8 мс | 505,9 мс | ~2726 мкс |
| ML-KEM-1024 (инкапсуляция + декапсуляция) | 39,6 мс | 33,8 мс | 244 мкс |
| **отношение** | **13,4 : 1** | **15,0 : 1** | **11,2 : 1** |

Результат устойчив: преимущество ML-KEM сохраняется и там, где железо помогает
конкуренту.

**Наблюдение 8. Аппаратный ускоритель ESP32-C6 разгоняет операции ECDSA, но не
обмен ключами ECDHE.** Разложение рукопожатия по двум измеренным режимам
позволяет разделить вклад:

| Составляющая | ESP32-S3 | ESP32-C6 | Ускорение |
|--------------|----------|----------|-----------|
| Отдельная подпись ECDSA | 170,2 мс | 22,1 мс | 7,7 × |
| Часть ECDSA в рукопожатии (разность режимов) | 1024,2 мс | 143,2 мс | 7,2 × |
| Обмен ECDHE | 530,8 мс | 505,9 мс | **1,05 ×** |

Ускорение части ECDSA внутри рукопожатия (7,2) практически совпадает с
ускорением отдельной подписи (7,7), тогда как ECDHE не ускоряется вовсе. Это
согласуется с устройством ускорителя: ESP-IDF предоставляет аппаратные
реализации подписи и проверки ECDSA, но не умножения точки кривой в общем виде,
которым пользуется ECDHE.

Практическое следствие для проектирования: **стоимость ML-KEM предсказуема и
слабо зависит от кристалла** (39,6 против 33,8 мс — разница 1,17 раза), тогда
как стоимость ECDSA меняется в 7,7 раза в зависимости от наличия ускорителя.
Бюджет времени, рассчитанный для схемы на ML-KEM, переносится между платформами;
для схемы на классической эллиптической криптографии — нет.

**Ограничение этих замеров.** Полное рукопожатие LACERT на платах сопоставить с
DTLS не удалось: потребовалась бы стоимость проверки подписи на самом кристалле,
которой в микробенчмарке нет — в протоколе подпись проверяет шлюз, а не
устройство. Сравнение на платах ограничено обменом ключами; протоколы целиком
сопоставлены только на x86 (таблица выше).

#### Чего сравнение не охватывает

Числа выше сопоставляют только установление сессии и шифрование. За их пределами
остаются механизмы, которых в DTLS нет вовсе:

- **непрерывная ротация ключей** — DTLS 1.2 не предусматривает смены ключа в
  рамках сессии; для этого требуется новое рукопожатие со всей его стоимостью;
- **проверка целостности прошивки** — в DTLS отсутствует как понятие;
- **отзыв отдельного устройства** без влияния на остальные — в режиме PSK
  недостижим в принципе.

Соответственно, вывод корректно формулировать так: при сопоставимых гарантиях
аутентификации и свежести ключа LACERT устанавливает сессию быстрее и
дополнительно предоставляет механизмы, отсутствующие в DTLS, ценой увеличенного
объёма рукопожатия.

#### Воспроизведение

Исходные тексты обоих стендов — `dtls_bench.c` и `lacert_hs_bench.c`; команды
сборки приведены в комментариях к файлам. Сертификаты для режима ECDHE-ECDSA
создаются заранее:

```bash
openssl ecparam -name prime256v1 -genkey -noout -out srv.key
openssl req -new -x509 -key srv.key -out srv.crt -days 365 \
  -subj "/CN=lacert-dtls-bench" -sha256
openssl ecparam -name prime256v1 -genkey -noout -out cli.key
openssl req -new -x509 -key cli.key -out cli.crt -days 365 \
  -subj "/CN=lacert-device-1" -sha256
```

## 4. Методика измерения производительности шлюза

Нагрузочное тестирование проводилось на одном процессорном ядре с реальной
СУБД PostgreSQL 16. Нагрузка задавалась встроенным эмулятором устройств
(`LACERT_EMULATE_DEVICES`), который реализует тот же сетевой протокол, что и
физические платы.

Параметры нагрузки намеренно выбраны жёстче эксплуатационных:

| Параметр | При испытании | По умолчанию |
|----------|---------------|--------------|
| Интервал телеметрии | 2 с | 2 с |
| Интервал ротации ключа | 30 с | 300 с |
| Интервал проверки прошивки | 60 с | 1 ч |

Измерялись: загрузка процессора и объём резидентной памяти процесса шлюза
(`ps`), время ответа REST API (`curl`), число успешных рукопожатий, ротаций и
проверок целостности (счётчики `/api/v1/metrics`), число записей в БД.

## 5. Результаты: производительность шлюза

| Устройств | Загрузка CPU | Память | Время ответа `/api/v1/devices` | Сбоев |
|-----------|--------------|--------|-------------------------------|-------|

Во всех испытаниях все устройства успешно устанавливали защищённое соединение;

Потребление ресурсов растёт линейно: приблизительно 0,025 % процессорного ядра и

## 6. Воспроизведение результатов

### 6.1. Криптографические измерения на плате

```bash
cd firmware
# в main/main.c задать параметры сети и LACERT_DEVICE_ID
rm -rf build sdkconfig
idf.py set-target esp32c6          # либо esp32s3
idf.py build

# убедиться, что применена оптимизированная конфигурация:
grep -E 'COMPILER_OPTIMIZATION|MBEDTLS_HARDWARE' sdkconfig

idf.py -p /dev/ttyACM0 flash monitor
```

Результаты выводятся в последовательный порт сразу после инициализации, до
подключения к сети (блок «МИКРОБЕНЧМАРК КРИПТОГРАФИИ»).

### 6.2. Измерения на базовой платформе (x86-64)

```bash
cd firmware/linux-debug
gcc -O2 -std=c11 -DBLAKE3_NO_SSE2 -DBLAKE3_NO_SSE41 -DBLAKE3_NO_AVX2 \
    -DBLAKE3_NO_AVX512 -I. -I../main -I../components/ml_kem -I../components/blake3 \
    bench_host.c ../main/lacert_crypto.c ../components/ml_kem/*.c \
    ../components/blake3/blake3.c ../components/blake3/blake3_dispatch.c \
    ../components/blake3/blake3_portable.c -lmbedcrypto -o bench_host
./bench_host
```

### 6.3. Нагрузочное тестирование шлюза

```bash
export LACERT_PG_DSN="postgresql://lacert:ПАРОЛЬ@localhost/lacert?sslmode=disable"
export LACERT_ADMIN_TOKEN=ТОКЕН
export LACERT_EMULATE_DEVICES=500
export LACERT_EMULATE_INTERVAL=2s
export LACERT_ROTATION_INTERVAL=30s
export LACERT_FIRMWARE_INTERVAL=60s
./gatewayd

# в другом терминале, спустя ~1 минуту:
ps -p $(pgrep -x gatewayd) -o %cpu=,rss=
curl -s -o /dev/null -w "%{time_total}\n" -H "Authorization: Bearer ТОКЕН" \
     http://localhost:8080/api/v1/devices
curl -s -H "Authorization: Bearer ТОКЕН" http://localhost:8080/api/v1/metrics
```

## 7. Первичные данные

Вывод бенчмарка сохраняется в журнале последовательного порта; протоколы
измерений по каждой плате приведены в приложении к работе. Значения телеметрии,

сохраняются в базе данных (поля `handshake_us`, `rotation_us`, `fw_sign_us`,
`heap_free`, `heap_min`) и доступны через `GET /api/v1/telemetry`, что позволяет
проверить приведённые значения независимо.
