# LACERT — прошивка ESP32 (клиент протокола)

Клиент протокола LACERT на C для ESP-IDF. **Один проект собирается под все три
платы**: XIAO ESP32-C6 (RISC-V), XIAO ESP32-S3 и ESP32-S3-N8R2 (Xtensa).
Разница архитектур скрыта внутри ESP-IDF — отдельного кода под плату не нужно.

Формат протокола — байт-в-байт по `PROTOCOL_SPEC.md`.

## Статус: ГОТОВО К ЗАЛИВКЕ

Все части реализованы, заглушек не осталось:

| Файл | Что делает | Статус |
|------|-----------|--------|
| `main/lacert_wire.c` | кадры и поля протокола | готов |
| `main/lacert_client.c` | рукопожатие, данные, ротация, проверка прошивки | готов |
| `main/lacert_crypto.c` | ECDSA, ChaCha20, SHA-256 (mbedTLS) + ML-KEM, BLAKE3 | готов |
| `main/main.c` | WiFi, ключи в NVS, регистрация, цикл, переподключение | готов |
| `component-overlay/` | обвязка компонентов под ESP32: ГСЧ (`randombytes_esp.c`), `CMakeLists.txt` для ml_kem и blake3 | в репозитории |
| `components/ml_kem/`, `components/blake3/` | ML-KEM-1024 (PQClean) и BLAKE3 — **скачиваются** скриптом `fetch-components.sh`, в репозитории не хранятся | см. «Сборка» |

### Проверено до заливки

Криптомодуль ESP32 (`lacert_crypto.c` + оба компонента) собран на хосте против
настоящих заголовков mbedTLS 3.6 (той же версии, что в ESP-IDF) и прогнан
против **живого Go-шлюза**. Результат: регистрация, рукопожатие, телеметрия,
**3 ротации ключа** и **5 проверок целостности прошивки** — всё принято шлюзом.
Значит, крипто и протокол совместимы; на плате остаётся только «обвязка»
(WiFi/NVS), которая тоже написана.

Дополнительно: полная сборка `idf.py build` под esp32c6 проверена из чистого
клона репозитория (после `fetch-components.sh`) — `lacert_firmware.bin`
создаётся и укладывается в раздел приложения.

## Перед сборкой: настроить стенд

В `main/main.c` вверху заполнить:

```c
#define LACERT_WIFI_SSID      "имя_вашей_сети"
#define LACERT_WIFI_PASS      "пароль"
#define LACERT_GW_HOST        "192.168.1.10"   // IP шлюза в локальной сети
#define LACERT_GW_HTTP_PORT   8080
#define LACERT_GW_TCP_PORT    7700
#define LACERT_DEVICE_ID      "xiao-esp32-1"   // уникальный на каждую плату!
#define LACERT_ADMIN_TOKEN    ""               // токен шлюза, если включён

#define LACERT_LED_MODE       1    // 1 = простой светодиод (XIAO)
                                   // 2 = адресный RGB WS2812 (ESP32-S3-DevKitC-1)
```

**Про светодиод.** У XIAO обычный одноцветный светодиод (режим 1), у
ESP32-S3-DevKitC-1 — адресный RGB WS2812 (режим 2), который управляется
импульсным протоколом и обычным `gpio_set_level` не зажигается. Поставьте режим
под свою плату. Пины (`LACERT_LED_GPIO` для XIAO, `LACERT_RGB_GPIO` для
DevKitC-1) заданы там же и при необходимости меняются.

Что показывает светодиод:

| Событие | XIAO (одноцветный) | DevKitC-1 (RGB) |
|---------|--------------------|-----------------|
| защищённый канал установлен | 3 вспышки | зелёный ×3 |
| передача телеметрии | 1 короткая | синий ×1 |
| ротация ключа | 2 вспышки | фиолетовый ×2 |
| проверка целостности прошивки | 1 вспышка | бирюзовый ×1 |
| ошибка / шлюз недоступен | длинная | красный |

**Важно:** у каждой платы должен быть свой `LACERT_DEVICE_ID`, иначе они будут
конфликтовать на шлюзе.

## Сборка и заливка

Сторонняя криптография (ML-KEM из PQClean, BLAKE3) не хранится в репозитории —
её нужно один раз скачать. Скрипт клонирует их на зафиксированных версиях,
отбирает только файлы, собираемые под ESP32 (SIMD-реализации под x86/ARM
отбрасываются), и накладывает локальную обвязку из `component-overlay/`
(аппаратный ГСЧ и `CMakeLists.txt`).

Команды ниже записаны для монорепозитория `lacert`, где прошивка лежит в
подкаталоге `firmware/`. Если вы собираете из репозитория
`lacert-crypto-esp32`, прошивка там находится в корне — шаг `cd firmware`
пропустите, остальные команды те же.

```bash
cd firmware

# 1. скачать сторонние компоненты (один раз; требует git и доступа в сеть)
./scripts/fetch-components.sh

# 2. выбрать плату (один раз):
idf.py set-target esp32c6      # XIAO ESP32-C6
#   или
idf.py set-target esp32s3      # XIAO ESP32-S3 / ESP32-S3-N8R2

# 3. собрать и залить
idf.py build
idf.py -p /dev/ttyUSB0 flash monitor    # порт под вашу систему
```

Версии PQClean и BLAKE3 зафиксированы в `scripts/fetch-components.sh`
(переменные `PQCLEAN_REF`, `BLAKE3_REF`) — сборка воспроизводима. Обновлять их
осознанно, меняя эти значения и проверяя сборку.

## Что происходит при первом включении

1. Поднимается WiFi.
2. Ключи в NVS не найдены → генерируются ECDSA P-256 + ML-KEM-1024 и
   сохраняются (аналог efuse: при следующих включениях те же).
3. Считается SHA-256 образа прошивки.
4. Устройство регистрируется на шлюзе через REST (повторная регистрация с теми
   же ключами — не ошибка, шлюз просто узнаёт устройство).
5. Забирается публичный ML-KEM-ключ шлюза.
6. TCP-подключение, рукопожатие, дальше цикл: телеметрия каждые 2 секунды,
   ответы на ротацию ключа и проверку прошивки. При разрыве — переподключение.

Ожидаемый лог в мониторе:

```
I (…) lacert: WiFi подключён, IP: 192.168.1.55
I (…) lacert: первый старт: генерирую ключи (ECDSA + ML-KEM-1024)...
I (…) lacert: ключи сгенерированы и сохранены в NVS
I (…) lacert: регистрация: новое устройство (201)
I (…) lacert: публичный ML-KEM шлюза получен
I (…) lacert: === защищённый канал установлен ===
I (…) lacert: отправлено: temperature=20.5; seq=0
I (…) lacert: [прошивка] ответил на проверку целостности
I (…) lacert: [ротация] новый ключ, итерация=1
```

## Отладка без платы

Каталог `linux-debug/` собирает тот же клиент под Linux (крипто на OpenSSL) для
проверки протокола против шлюза без железа — полезно при доработках.

## Память

ML-KEM-1024 требует заметного стека (ключ 3168 байт + рабочие буферы).
В `sdkconfig.defaults` стек главной задачи поднят до 24 КБ. На ESP32-C6 памяти
меньше, чем на S3 — если появятся сбои выделения, включите PSRAM на S3-N8R2 или
уменьшите буферы.

## Подпись

Используется **ECDSA P-256** — по результатам анализа (SLH-DSA на ESP32
непрактичен: подпись сотни миллисекунд и 7856 байт).

Аппаратный ускоритель эллиптических кривых есть **только у ESP32-C6**: там
подпись занимает 22,2 мс. У ESP32-S3 такого блока нет, и та же операция
выполняется программно за 170,2 мс — разница в 7,7 раза. Это стоит учитывать
при выборе платы, если рукопожатия происходят часто.


## Частые проблемы при сборке и заливке

Собрано по опыту реального развёртывания на Manjaro/Arch.

**`idf.py: command not found`.** Окружение ESP-IDF не активировано. В каждой
новой сессии терминала:

```bash
. ~/esp/esp-idf/export.sh
```

**Список целей (`set-target`) пуст.** ESP-IDF не установлен или расширение
VS Code не знает, где он лежит. Проверьте `echo $IDF_PATH`.

**Порт не открывается (`Permission denied`).** В Arch/Manjaro группа для
последовательных портов — `uucp` (а не `dialout`, как в Ubuntu):

```bash
sudo usermod -aG uucp $USER    # затем перелогиниться
```

**Какой порт у платы.** XIAO использует нативный USB чипа и появляется как
`/dev/ttyACM0`. У ESP32-S3-DevKitC-1 стоит внешний USB-UART мост, поэтому она
появляется как `/dev/ttyUSB0`. Если esptool не находит порт — проверьте оба.

**Плата не отвечает / порт исчез / `error -71`.** Нативный USB у XIAO может
«залипнуть» после неудачной попытки сброса. Ручной вход в загрузчик: отключить
USB → зажать и держать **BOOT** → воткнуть USB → отпустить BOOT. После этого
`idf.py -p /dev/ttyACM0 flash`, затем нажать RESET.

**`build.ninja: expected build command name`.** Каталог сборки повреждён после
прерванной сборки. Лечится `rm -rf build`.

**Сборка падает внутри ESP-IDF (internal compiler error, `corrupted size`).**
Признак повреждённого кеша или нестабильного состояния ветки IDF. Проверьте,
что стоит релизный тег, а не промежуточный коммит:

```bash
cd ~/esp/esp-idf && git describe --tags      # должно быть, напр., v5.4.4
```

Если хвост вида `v5.4.4-813-g...` — переключитесь на тег:
`git checkout v5.4.4 && git submodule update --init --recursive && ./install.sh esp32s3,esp32c6`.
Также помогает `ccache -C` и сборка в один поток: `idf.py -j1 build`.

**`undefined reference to mbedtls_chachapoly_*`.** В `sdkconfig` не включён
ChaCha20-Poly1305. Опция `CHACHAPOLY` зависит от `POLY1305`, а та — от
`CHACHA20`, поэтому нужны все три (они есть в `sdkconfig.defaults`). Если
`sdkconfig` уже создан, правки в defaults не подхватятся — удалите его и
пересоздайте: `rm -f sdkconfig && idf.py set-target esp32s3`.

**Выход из монитора** — `Ctrl+]`. Пересобрать и перезалить, не выходя из
монитора: `Ctrl+T`, затем `Ctrl+F`.


## Таблица разделов

Прошивка не помещается в штатный раздел приложения (1 МБ): ML-KEM + BLAKE3 +
mbedTLS + WiFi дают около 1,0–1,1 МБ. Поэтому в проекте своя таблица
(`partitions.csv`) с разделом приложения на **2 МБ** — она подключена через
`sdkconfig.defaults` и применяется автоматически.

Флеш плат: XIAO ESP32-C6 — 4 МБ, XIAO ESP32-S3 и ESP32-S3-DevKitC-1 — 8 МБ.
Настройка `CONFIG_ESPTOOLPY_FLASHSIZE_4MB` подходит всем (на 8-мегабайтных
платах просто не используется остаток).


## Замеры производительности на плате

В `main/main.c` есть два независимых механизма (оба включены по умолчанию):

**`LACERT_MEASURE`** — измеряет реальные операции протокола (рукопожатие,
ротацию, ответ на проверку прошивки) и отправляет результаты вместе с
телеметрией. Они попадают в базу и на графики дашборда. Полезно для контроля
работы и поиска утечек памяти (поля `heap_free`, `heap_min`).

Важно: эти цифры **включают сетевой обмен** по WiFi, поэтому для сравнения
платформ они не годятся.

**`LACERT_BENCH`** — микробенчмарк: при старте (до подключения к WiFi) прогоняет
каждую криптооперацию 20 раз **без сети** и печатает средние значения:

```
==================================================
МИКРОБЕНЧМАРК КРИПТОГРАФИИ (без сети)
плата: ESP32-C6 (RISC-V, 160 МГц)
--------------------------------------------------
  ECDSA P-256 keypair              9.63 мс   (5 прогонов)
  ECDSA P-256 sign                22.20 мс   (20 прогонов)
  ML-KEM-1024 keypair             15.06 мс   (5 прогонов)
  ML-KEM-1024 encapsulate         15.99 мс   (20 прогонов)
  ML-KEM-1024 decapsulate         17.78 мс   (20 прогонов)
  BLAKE3 (вывод ключа)             19.8 мкс  (200 прогонов)
  SHA-256 (96 байт)                41.7 мкс  (200 прогонов)
  ChaCha20-Poly1305 seal          210.6 мкс  (200 прогонов)
==================================================
```

Это и есть корректный инструмент для сравнения архитектур. Тот же набор операций
тем же кодом гоняется на хосте — `linux-debug/bench_host.c`.

Результаты по всем платам и выводы — в `FIRMWARE.md`.

**Замеры зависят от настроек сборки.** По умолчанию ESP-IDF собирает с `-Og`
(без оптимизации) — на такой сборке криптография работает в разы медленнее.
В `sdkconfig.defaults` включены `-Os` и аппаратные ускорители (MPI, SHA, а для
C6 — ECC). Проверить, что они применились:

```bash
grep -E 'COMPILER_OPTIMIZATION|MBEDTLS_HARDWARE' sdkconfig
```

Если правили `sdkconfig.defaults`, старый `sdkconfig` правки не подхватит — его
надо удалить: `rm -f sdkconfig && idf.py set-target esp32s3`.
